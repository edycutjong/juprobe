## Endpoint-by-Endpoint Analysis

### 1. Quote API (`v6.jup.ag/quote`)

**DX Grade: A-** (85/100)

| Metric | Value |
|---|---|
| p50 Latency | 124ms |
| p95 Latency | 287ms |
| p99 Latency | 412ms |
| Error Rate | 0.3% |
| Rate Limit Threshold | ~600 req/min (estimated) |

**Strengths:**
- Fastest endpoint in the Jupiter suite — sub-200ms p50 is excellent for real-time quoting
- Response payload is well-structured with `routePlan` breakdowns
- The `slippageBps` parameter is intuitive (basis points, not decimals)
- Automatic route optimization across 20+ AMMs

**Friction Points:**
- **Silent 429s**: When rate limited, the endpoint returns a blank response body with a 429 status. No `Retry-After` header, no error message, no rate limit headers. A developer debugging this for the first time will waste hours.
- **Amount precision**: The `amount` parameter requires raw lamport/token amounts (e.g., `1000000` for 1 USDC). While standard for Solana, the SDK docs don't prominently state this — many devs pass `1` expecting 1 USDC.

**Recommendation:**
```
Add standard rate limit headers:
  X-RateLimit-Limit: 600
  X-RateLimit-Remaining: 42
  X-RateLimit-Reset: 1716480000

Return a JSON error body on 429:
  { "error": "rate_limited", "retryAfterMs": 2000 }
```

### 2. Swap API (`v6.jup.ag/swap`)

**DX Grade: B** (78/100)

| Metric | Value |
|---|---|
| p50 Latency | 854ms |
| p95 Latency | 1,420ms |
| p99 Latency | 2,100ms |
| Error Rate | 4.2% |
| Rate Limit Threshold | ~120 req/min (estimated) |

**Strengths:**
- Transaction construction is fully server-side — developers don't need to manually build Solana transactions
- `wrapAndUnwrapSol` convenience parameter eliminates wSOL management boilerplate
- The `swapResponse.swapTransaction` is a ready-to-sign base64 transaction

**Friction Points:**
- **High p95 tail latency**: The 1.4s p95 means 1 in 20 swap requests takes over a second. For time-sensitive DeFi applications (arbitrage, liquidation), this is problematic.
- **Opaque error messages on invalid public keys**: Passing an invalid `userPublicKey` returns `{"error":"Failed to deserialize account"}` — no indication that the issue is the key format, not a blockchain state error.
- **No partial failure reporting**: When a multi-hop route partially fails, the error doesn't indicate which hop failed.

**Recommendation:**
```
Validate `userPublicKey` format before processing:
  { "error": "invalid_public_key", "field": "userPublicKey", 
    "expected": "base58 Solana public key (32 bytes)" }

Add hop-level error attribution for multi-route swaps:
  { "error": "route_partial_failure", "failedHop": 2, 
    "pool": "Raydium SOL/USDC", "reason": "insufficient_liquidity" }
```

**Friction Points:**
- **Webhook documentation gap**: The `callbackUrl` parameter exists but isn't documented in the API reference. Developers discover it through community Discord, not docs.

**Recommendation:**
```
Document the webhook callback parameter:
  POST /limit/v2/createOrder
  Body: { ..., "callbackUrl": "https://yourapp.com/webhook/jupiter" }
  Webhook payload: { "orderId": "...", "status": "filled", "filledAt": 1716480000 }
```

### 4. DCA (Dollar Cost Averaging) API (`jup.ag/dca/v1`)

**DX Grade: C+** (67/100)

| Metric | Value |
|---|---|
| p50 Latency | 1,205ms |
| p95 Latency | 2,800ms |
| p99 Latency | 4,100ms |
| Error Rate | 8.7% |
| Rate Limit Threshold | ~60 req/min (estimated) |

**Strengths:**
- DCA is a complex on-chain program — the API abstracts significant complexity
- Flexible interval configuration (minutely to monthly)

**Friction Points:**
- **Highest error rate** across all endpoints (8.7%). Most errors are silent timeouts with no error body.
- **Extreme tail latency**: p99 of 4.1s means ~1% of requests take over 4 seconds. No timeout guidance in docs.
- **Missing idempotency keys**: Creating a DCA order twice (e.g., due to network retry) creates duplicate orders. No deduplication mechanism.
- **Undocumented minimum amounts**: Some token pairs silently reject DCA orders below an undocumented minimum — no error, just a 200 OK with no order created.

**Recommendation:**
```
1. Add idempotency key support:
   Header: X-Idempotency-Key: <uuid>

2. Return explicit errors for minimum amount violations:
   { "error": "below_minimum", "minAmount": 10000000, 
     "currency": "USDC", "humanReadable": "10 USDC" }

3. Document recommended timeout values:
   "DCA order creation: set client timeout to ≥5000ms"
```

- Fast CDN-backed response times
- Comprehensive token metadata (name, symbol, decimals, logoURI, tags)

**Friction Points:**
- **Large payload size**: The full token list returns 15,000+ tokens (~4MB uncompressed). No pagination or cursor-based loading.
- **No delta/diff endpoint**: Applications that poll for new tokens must re-download the entire list. A `GET /tokens?since=<timestamp>` would save bandwidth.

**Recommendation:**
```
Add pagination:
  GET /tokens?page=1&limit=100&verified=true

Add delta endpoint:
  GET /tokens/updates?since=2026-05-01T00:00:00Z
  Response: { "added": [...], "removed": [...], "updated": [...] }
```

### 6. Price API (`price.jup.ag/v4`)

**DX Grade: A-** (84/100)

| Metric | Value |
|---|---|
| p50 Latency | 62ms |
| p95 Latency | 130ms |
| p99 Latency | 210ms |
| Error Rate | 0.5% |
| Rate Limit Threshold | ~900 req/min (estimated) |

**Strengths:**
- Fast and reliable for single-token lookups
- Batch pricing endpoint (`/price?ids=SOL,USDC,JUP`) reduces round trips
- USD and SOL denomination options

**Friction Points:**
- **Batch limit undocumented**: Requesting prices for 100+ tokens in one call silently drops tokens beyond an undocumented limit (~50). No error, no warning — the response just has fewer entries than requested.
- **Stale price indicators missing**: No `lastUpdatedAt` timestamp in the response. Developers can't distinguish between a live price and a cached/stale price.
- **Inconsistent error for unknown tokens**: Requesting a price for a non-existent mint address returns `{}` (empty object) instead of a proper error.

**Recommendation:**
```
Add staleness metadata:
  { "SOL": { "price": 168.42, "lastUpdatedAt": 1716480000, 
    "confidence": "high" } }

Return explicit errors for unknown tokens:
  { "SOL": { "price": 168.42 }, 
    "UNKNOWN_MINT": { "error": "token_not_found" } }

Document batch limits:
  "Maximum 50 token IDs per request. Excess IDs are silently dropped."
```

## Cross-Cutting Concerns

### Rate Limiting
**Grade: D** (55/100)

The single biggest DX issue. No endpoint returns rate limit headers. No documentation states the limits. Developers must reverse-engineer thresholds through trial and error.

| Endpoint | Estimated Limit | Documented? |
|---|---|---|
| Quote | ~600/min | ❌ |
| Swap | ~120/min | ❌ |
| Limit | ~300/min | ❌ |
| DCA | ~60/min | ❌ |
| Token List | ~1200/min | ❌ |
| Price | ~900/min | ❌ |

**Impact**: A DeFi developer building a portfolio tracker that polls prices for 200 tokens will unknowingly hit the rate limit within seconds. No warning, no degradation — just silent failures.

### Error Message Consistency
**Grade: C+** (68/100)

Error formats vary across endpoints:
- Quote: `{ "error": "string message" }`
- Swap: `{ "error": "string message" }` (but different field names)
- Token List: HTTP status only, no body
- Price: Empty object `{}`

**Recommendation**: Adopt a unified error schema:
```json
{
  "error": {
    "code": "RATE_LIMITED",
    "message": "Too many requests. Retry after 2000ms.",
    "retryAfterMs": 2000,
    "endpoint": "/v6/quote",
    "requestId": "req_abc123"
  }
}
```
